# Composition Hallucination Benchmark

A benchmark for evaluating **composition hallucination** in retrieval-augmented generation (RAG): failures where every necessary fragment is present in context and each fragment is locally legible, but the model answers incorrectly because it fails to compose the *relations* among fragments — exceptions, overrides, scopes, preconditions, sequences, contraindications, temporal dependencies, and exception-of-exception structures.

**Authors:** CKF Research; Paulo Tomazinho.
**Version:** `0.5.0`.
**License:** CC BY 4.0 (see `LICENSE` for the adapted-real source caveat).

The benchmark implements the protocol in *Composition Hallucination in Retrieval-Augmented Generation: A Failure Mode and Benchmark Protocol* (Tomazinho, CKF Research, 2026), included at `docs/composition_hallucination_protocol_paper_2026.pdf`. The protocol requires evidence sufficiency, local-legibility probes, counterfactual explicitness, position controls, retrieval controls, and metadata for relation type, domain, and complexity.

---

## The phenomenon in one example

A corporate policy says employees may approve expenses up to \$5,000. Another section says international expenses require CFO approval regardless of amount. Asked "Can I approve an international expense of \$3,000?", a RAG system retrieves both clauses, reads each correctly, and answers: "Yes, it's under \$5,000." The evidence was present and legible; the model failed to compose the exception. That is composition hallucination — distinct from retrieval failure, parametric hallucination, and long-context (positional) failure.

## What's in this release

- **40 canonical English cases** (`cases/`): 20 synthetic (`01`–`20`) and 20 adapted-real (`21`–`40`) derived from public source documents (Kubernetes, AWS, DailyMed labels, IRS guidance, university policies, RFCs).
- **40 Portuguese-Brazilian draft translations** (`cases_pt-br/`) for bilingual review and language-dependence experiments.
- **40 experimental CKF-compatible exports** (`ckf/cases/`) for structured-knowledge / GraphRAG / agentic experiments.
- **A Python evaluation runner** (`scripts/`) — new in this release — that runs the cases against Anthropic, OpenAI, and Google models, classifies responses, and produces per-response and aggregate output. Includes an offline `mock` provider so the full pipeline can be exercised with no API key and no cost.
- **Derived data artifacts** (`data/`) for HuggingFace loading and quick inspection, all regenerable from the canonical cases.

The canonical source of truth is `cases/*.json`, validated by `schema/case_schema.json`. Everything in `data/`, `ckf/`, and `cases_pt-br/` is a parallel or derived resource.

## Case format

Each case is a JSON object with:

- `metadata`: domain, relation type, complexity level, source status, license, notes.
- `provenance` (adapted-real cases): source title, organization, URL, access date, adaptation method, professional-guidance warning.
- `representations.raw`: natural prose, relations implicit.
- `representations.annotated`: same content with inline relation tags.
- `representations.compiled`: same content as typed, structured (YAML-like) objects.
- `query`: the composition-sensitive question.
- `gold_answer`: full answer, minimal answer, and the list of required relations.
- `local_probes`: fragment-level checks for local legibility (protocol §7.2).
- `failure_examples`: labeled wrong-answer patterns (§8.3).
- `controls`: necessary fragments, negative-control query and gold, position-variant support, parametric-contamination risk.

The three representation conditions hold informational content constant and vary only relational explicitness. If accuracy rises from `raw` to `annotated` to `compiled`, that difference is the quantity the benchmark measures. The audit confirming content equivalence is in `CONTENT_EQUIVALENCE_AUDIT.md`.

## Quick start: running the benchmark

```bash
pip install -r scripts/requirements.txt        # install only the SDKs you use
export ANTHROPIC_API_KEY=...                    # set keys for the providers you run

# 1. Validate every case against the schema and check EN/PT/CKF parity
python scripts/validate_cases.py

# 2. (Re)build derived data artifacts from the canonical cases
python scripts/build_jsonl.py --version 0.5.0

# 3. Estimate cost of a run without calling any API
python scripts/run_eval.py --models anthropic:claude-opus-4-7 --dry-run

# 4. Exercise the whole pipeline offline, no cost, no key
python scripts/run_eval.py --models mock:no --output-dir runs/mock --yes

# 5. A real run: one model, all cases, all conditions, all controls, out-of-family judge
python scripts/run_eval.py \
    --models anthropic:claude-opus-4-7 \
    --judge-model openai:gpt-4o-2024-08-06 \
    --conditions raw annotated compiled \
    --query-types primary negative_control local_probe \
    --output-dir runs/opus-full/

# 6. Re-score an existing run (e.g. add a judge after a cheap heuristic-only run)
python scripts/score_outputs.py runs/opus-full/results.jsonl \
    --judge-model openai:gpt-4o-2024-08-06
```

The runner enforces the protocol's controls: it holds retrieval constant (the full source is always in context), supports the primary query, the negative control, and local probes, and **blocks using an LLM-as-judge from the same model family as the evaluated model** (protocol §8.4). Full runner documentation is in `scripts/README.md`.

## How to falsify the hypothesis

This benchmark exists to test a falsifiable claim and to make refuting it easy. The most valuable contribution is a credible attempt to break it. When you open a run's `summary.csv`:

1. **Does accuracy rise from `raw` to `annotated` to `compiled`**, content held constant? That gap is the phenomenon. No gap (after controlling for position and retrieval) is evidence *against* the representational hypothesis — please report it.
2. **Does the gap widen with `complexity_level`?** The protocol predicts it should; level-3 cases are where this is tested. A flat or shrinking gap refines or contradicts the prediction.
3. **Do local probes pass?** If a model fails a local probe, its primary-query failure is comprehension, not composition.
4. **Are the negative controls right?** High primary accuracy with low negative-control accuracy means the model is defaulting conservatively, not composing. The `wrong_escalation` count is a related signal.
5. **Do heuristic and judge labels agree?** Low agreement means noisy scoring — hand-review before drawing conclusions.

Other strong falsifications: show a case where the annotated/compiled condition silently adds information (a confound — see `CONTENT_EQUIVALENCE_AUDIT.md`); show that the failure reduces to a known mode; show that a simple prompt scaffold closes the gap (compare `--prompt-style standard` vs `minimal`).

## Intended evaluation use

The primary composition test uses perfect retrieval: all `controls.necessary_fragments` are present in context. A response counts as a composition error only when evidence is present, local probes are answerable, and the final answer is wrong because a required relation was missed, misapplied, or prioritized incorrectly. For adapted-real cases, score against the benchmark case text, not the live source document; provenance URLs are for traceability and audit, not dynamic retrieval. See `docs/SCORING.md`.

## Ethical and domain notes

Adapted-real cases are evaluation artifacts. Clinical, legal, tax, regulatory, cloud-operations, immigration, housing, and university-policy cases are **not professional guidance** and must not be used for real decisions. Real-world deployment or publication should involve domain experts and source-license review. See the ethics and availability notes in the protocol paper and `LICENSE`.
